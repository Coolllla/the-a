# 2026-07-24 路由转场动画调研：结论是先不做

## 背景

研究"路由跳转时的转场动画"怎么做。本版 Next.js（16.2.6）内置了基于 React 19 `<ViewTransition>` 的 View Transitions API 集成（`experimental.viewTransition: true`），官方指南在 `node_modules/next/dist/docs/01-app/02-guides/view-transitions.md`，提供四种模式：共享元素 morph / Suspense reveal / 方向性滑动（`<Link transitionTypes>`）/ 同路由 crossfade。

## 关键认知：转场 ≠ 开屏

一次跳转拆成两段：

1. **切换瞬间**（View Transition 的地盘）：对旧页面截图，在截图与新页面之间播 <500ms 的 CSS 动画。因为操作对象是截图，做不了 JS 驱动的复杂演出。职责只是让"剪辑点"不生硬。
2. **新页面落地后**（开屏动画的地盘）：活的 DOM，GSAP timeline 自由演出——幕布、logo、错峰入场、3D、canvas 都行。游戏活动页式的开屏本质全是这一段，与路由 API 无关。

本站想要的"炫酷进场"效果属于第 2 段，由既有技术栈（tech-stack.md 3.5：场景级 → GSAP）覆盖，不需要 View Transition。

## 为什么决定先不用 View Transition

- 硬切是 web 默认体验，普通页感知很低；重点页有专门的入场演出承接。
- 动画设计哲学是**由目标页决定、与出发页无关**，共享元素 morph（两页连续性）无用武之地。
- App Router 客户端导航本身就是"旧页保持可见 → 新路由就绪才换屏"+ prefetch，加载空窗已被 Next 垫掉。
- 旧页面退场演出暂无需求。
- 功能是 experimental（React 组件 API 可能变），能绕开先绕开。

**推迟代价≈0**：这套是纯增量（flag + 包一层 + CSS），无架构预埋。未来出现"列表→详情连续感"或退场需求时随时补，已写的开屏动画不受影响。

## 将来要捡起来时的速查

- `next.config.ts` 开 `experimental: { viewTransition: true }`；`import { ViewTransition } from 'react'`。
- 路由跳转自动触发；`default="none"` 防止无关转场互相牵连。
- Nav 要钉住：`viewTransitionName: 'site-header'` + CSS 关掉其动画（老快照 `display: none` 防重影）。
- 浏览器后退按钮不携带 transitionTypes，方向滑动只对站内点击生效。
- 官方节奏参数：退场 150ms、入场 210ms 且延迟至退场结束、位移 60px；记得 `prefers-reduced-motion` 归零。
- Demo：https://react-view-transitions-demo.labs.vercel.dev

## 开屏动画路线的注意点

- 入场元素的**初始隐藏态写在 CSS 里**（opacity 0 / 位移出场外），不能等 GSAP 挂载后再藏，否则终态闪现一帧（FOUC）。
- 旧页面"先演退场再跳转"若真要做：要么 View Transition exit 关键帧对截图做剪辑感处理（便宜），要么拦截导航跑完 timeline 再 `router.push`（自由但复杂），按页单独立项，不做全局机制。
